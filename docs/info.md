# [PRD] 작가를 위한 로컬 AI 기반 올인원 집필 스튜디오, '노덱스 스튜디오(Nodex Studio)'

## 1. 프로젝트 개요 (Product Overview)

### 1.1 비전·한 줄 정의
* **Product Vision:** "검색은 AI에게 맡기고, 작가는 오직 글쓰기에만 집중하는 환경을 만든다."
* **궁극 비전 (North Star):** 작가가 쌓아 올린 **원고·설정·대사**를 바탕으로, 장기적으로 **웹툰(컷 구성·캐릭터·배경 등)까지 자동 또는 반자동으로 생성·편집**할 수 있는 흐름을 목표로 한다. (초기 MVP·Phase 1~4에서는 **텍스트 집필·로어 RAG**에 집중하고, 웹툰 파이프라인은 별도 후속 로드맵으로 설계한다.)
* **한 줄 정의 (현 단계 제품):** 로컬 폴더·클라우드 백업·벡터 검색(RAG)을 하나의 **분할 화면 집필 앱**에 묶고, 생성은 **WebLLM(로컬)** 과 **Google Gemini API(선택)** 로 나누어 쓰는 **웹소설·웹툰 스토리 작가** 특화 SaaS.

### 1.2 타깃 사용자
* **핵심 타깃:** **웹소설·웹툰(스토리·각본) 작가** 한 그룹으로 통일한다. 방대한 세계관 노트, 인물·아이템 설정, 회차·컷 단위 원고를 동시에 다루며, 미공개 원고 유출에 민감하고 외부 AI API 비용에 부담을 느낄 수 있음.
* **공통 행동 특성:** 집필 중 수시로 "이름·설정·대사·전개"를 찾아봄; 나중에는 같은 맥락이 **시각 연출(웹툰)** 으로 이어지길 원하지만, 당장은 **글 쓰기와 설정 일관성**이 최우선.

#### 1.2.1 페르소나 (구체화, 작가만)
| 페르소나 | **지민** (웹소설 연재 작가) | **수진** (웹툰 각본·스토리 작가) |
|-----------|------------------------------|----------------------------------|
| **목표** | 연재 일정을 지키면서 설정 오류 없이 회차를 완성 | 회·컷 단위로 대사·지문이 설정·이전 화와 맞는지 빠르게 검증 |
| **전형적 하루** | 원고 에디터에 3~5시간 몰입, 중간에 설정 노트·메모를 수십 번 검색 | 시나리오 문서와 설정집, 레퍼런스를 오가며 컷 설명을 다듬음 |
| **불만** | 노션·폴더·한글을 오가며 **문맥이 끊김**; 검색창에 뭐라 쳐야 할지 기억이 안 남 | 대사/지문이 **캐릭터 말투·외형 설정**과 어긋나기 쉬움; 나중에 그림 작업 넘길 때 설명이 부족 |
| **이 제품에서의 성공** | **한 화면**에서 쓰면서 곁눈질로 로어를 물어볼 수 있음 | 드래그·짧은 질문으로 **내 프로젝트 설정만** 근거로 답을 받고, (장기적으로) 텍스트 기반 **웹툰 시각화**로 이어지길 기대 |
| **기술 친화도** | 중간 — 브라우저·구글 로그인은 익숙, API 키는 부담스러울 수 있음 | 중간 — 스튜디오·에셋 툴 경험 있음, 집필 툴은 통합되길 원함 |

#### 1.2.2 사용자가 제품을 이해하는 방식 (멘탈 모델)
* **왼쪽 = 내가 쓰는 글**, **오른쪽 = 내 설정집을 읽고 답하는 조수** — 검색 엔진이 아니라 **내 폴더/프로젝트에 귀 기울이는 AI**라는 인상을 주어야 함.
* **WebLLM**은 "내 PC 안에서 도는 AI", **Gemini**는 "구글 서버로 보내는 선택 옵션"으로 라벨을 명확히 구분(설정 화면에 한 줄 요약 + 링크).
* **색인 중/동기화 중**은 "조수가 내 책장을 정리하는 중"에 비유할 수 있는 짧은 문구로 불안 완화.

### 1.3 해결하려는 문제 (Pain Point)
1. **컨텍스트 스위칭:** 한글, 노션, 로컬 폴더 등에 설정이 흩어져 있어 설정 확인만으로도 집필 흐름이 끊김.
2. **보안 및 비용:** 기존 LLM 보조 툴은 원고가 외부로 나갈 수 있고, 종량제 API는 장기 사용 시 부담이 큼.
3. **동기화·백업:** 한 PC에만 두면 분실·기기 변경에 취약하고, 클라우드만 쓰면 편집 경험이 분산됨.

### 1.4 솔루션 요약
* 브라우저 **WebGPU** 기반 **WebLLM**으로 기본 생성을 로컬에서 수행하고, 고품질·저지연 응답이 필요할 때만 **Google Gemini API** 경로를 선택할 수 있게 함.
* **내 설정집**만 대상으로 하는 **RAG**(PostgreSQL `pgvector`)로 검색·인용을 서버에서 처리.
* 원본·백업은 **MinIO**, 비동기 색인·무거운 작업은 **Redis + Celery**.
* 계정은 **Google OAuth 2.0** + **JWT**(액세스·리프레시)로 API 인증.

### 1.5 범위 밖 (Non-Goals, 초기 버전)
* 협업 실시간 공동 편집(동시에 여러 명이 한 문서 편집) — 우선 1인 집필·동기화에 집중.
* 범용 노션 대체(위키·DB·전체 생산성) — **집필·로어 참조**에 특화.
* 자체 호스팅 온프레미스 패키지 — 초기에는 SaaS 중심(추후 검토 가능).
* **텍스트→웹툰 자동 생성(이미지·컷 분할·툰 스튜디오 연동)** — §1.1 궁극 비전에 두되, **Phase 1~4 제품 범위에서는 제외**(별도 Phase·스펙으로 정의).

### 1.6 사용자 관점 성공 지표 (정성·정량)
* **정성:** "설정 찾느라 탭을 안 돌린다", "민감한 회차도 AI에 올리기 싫었는데 로컬로 해결" 같은 인터뷰 코멘트.
* **정량 (예시):** 주간 활성 시 **LoreDex 질의당 평균 소요 시간**, **에디터 포커스 유지 시간**(탭 이탈 전후 비교는 CBT에서 설문·로그 병행).
* **기술 체감:** RAG 응답에 **출처 카드**가 붙어 사용자가 "근거 확인"을 클릭하는 비율(환각 불신 완화 여부).

### 1.7 작품 제작 흐름 (사용자 관점 한눈에)
본 제품은 **한 작품 = 하나의 프로젝트**로 두고, 아래 순서를 전제로 설계한다.

1. **프로젝트 만들기** — 작품 제목·(선택) 장르·표기 규칙 등 메타를 두어 LoreDex·검색 맥락을 좁힌다.
2. **소설 설정집 구성** — 로컬 폴더를 연결하거나 파일을 올려 **인물·세계관·용어** 등을 `.txt`/`.md` 로 정리하고, 트리·색인으로 **내 설정집만** RAG에 태운다.
3. **편·권·회차 단위로 집필** — 원고 파일을 **권/편/회** 등 작가 규칙에 맞게 두고(폴더·파일명 컨벤션), 에디터에서 회차를 쓴다.
4. **집필 중 AI(LoreDex)** — 글을 대신 쓰는 것이 아니라, **설정 확인·말투·연속성·용어** 때문에 멈추는 시간을 줄이는 **조수**(RAG + WebLLM/Gemini)로 쓴다.

## 2. 핵심 가치 제안 (Core Value Proposition)

### 2.1 Privacy First (로컬 우선)
* **WebLLM**을 기본 생성 엔진으로 두면, 질문·RAG로 가져온 **컨텍스트 조합**은 브라우저 내에서 추론에만 쓰이고, 사용자가 Gemini를 켜지 않는 한 **Google 쪽으로 원고 전송이 발생하지 않음**(프론트엔드 직접 호출 또는 사용자 동의 하 백엔드 프록시 정책은 구현 단계에서 확정).
* **명시적 동의:** Gemini 사용 시 "이 요청은 클라우드로 전송됩니다" 같은 UI를 띄워 선택을 기록.

### 2.2 Dual AI (WebLLM + Gemini)
| 구분 | WebLLM | Gemini API |
|------|--------|------------|
| 실행 위치 | 사용자 브라우저(WebGPU) | Google 클라우드 |
| 비용 | 사용자 기기 자원, 서비스 과금 거의 없음 | 토큰·API 정책에 따름(티어·쿼터로 제한) |
| 적합한 경우 | 민감한 원고, 오프라인에 가까운 환경, 비용 최소화 | 긴 문맥 요약, 빠른 응답, 로컬 GPU 부족 시 |
| 데이터 흐름 | 프롬프트·RAG 컨텍스트가 브라우저 내 | API 요청에 포함되는 텍스트가 Google 정책 적용 |

* **운영 전략:** 서비스 차원의 **Gemini API 키**(서버에서 프록시) vs 사용자 **BYOK(Bring Your Own Key)** 는 BM·보안 검토 후 결정. PRD에는 두 방식 모두 **옵션**으로 열어 둠.

### 2.3 Local-Cloud Hybrid
* 로컬 **File System Access API** 로 `.txt` 등을 읽고, 변경분은 **Django → MinIO** 백업 후 **Celery**로 청크·임베딩·`pgvector` 업데이트.
* 다른 기기에서는 동일 계정으로 로그인 후 **동기화된 프로젝트**만 검색·편집(티어에 따라).

### 2.4 AI의 역할 (작가 대체가 아닌 ‘고민 감소’)
* **원칙:** AI는 **최종 문장의 저자를 바꾸지 않는다.** 기본 UX는 **질문·인용·제안 초안**이며, 삽입·수정은 항상 **작가가 확정**한다.
* **덜어 주는 고민의 유형:** (1) **설정 맞추기** — 내가 예전에 적어 둔 설정과 모순 없는지, (2) **기억·탐색** — 이름·아이템·지명을 빨리 찾기, (3) **톤·말투** — 캐릭터에 맞는지 점검·다듬기 초안, (4) **연속성** — 직전 회·이전 대사와의 정합, (5) **용어·표기** — 같은 말을 통일했는지, (6) **막힐 때** — 다음 전개 아이디어를 **참고용**으로만 제시.
* **RAG 스코프:** 질문 시 **설정집만** / **직전 N회 원고 포함** / **현재 파일만** 등을 선택할 수 있게 하면(§3.11) 환각·노이즈를 줄인다.

## 3. 핵심 기능 및 사용자 여정 (Key Features & User Journey)

### 3.1 화면·레이아웃 구성 (UI Shell & Layout)

#### 3.1.1 정보 구조 (IA) 요약
* **비로그인:** 랜딩(가치 제안) → **Google 로그인** CTA.
* **로그인 후:** **프로젝트 목록**(또는 마지막 프로젝트 자동 진입) → **집필 스튜디오(메인)** 가 핵심 화면.
* **부가:** 계정·**설정**(AI 엔진 기본값, Gemini 동의, BYOK, 테마), (Pro) **동기화·스토리지** 상태.

#### 3.1.2 메인 스튜디오: 프로젝트 파일 트리 + 집필 + LoreDex (3분할)
작가는 **설정집이 어떤 폴더 구조로 들어와 있는지** UI에서 바로 볼 수 있어야 한다. 메인 화면 기본 구성은 **좌: 프로젝트 파일·폴더 트리**, **중앙: Focus Editor**, **우: LoreDex** 이다. 트리 열은 **접기(아이콘만 표시)** 가능한 **사이드 레일**로 두고, 좁은 화면에서는 **오버레이·햄버거**로 동일 정보에 접근한다.

```
┌──────────────────────────────────────────────────────────────────────────────────┐
│ App Bar: 로고 · 프로젝트 선택 · 동기화/색인 상태 · AI 엔진(WebLLM/Gemini) · 설정 · 프로필 │
├───┬───────────────────────────────────────────────┬────────────────────────────┤
│ 프로젝트 │ Focus Editor (Tiptap)                    │ LoreDex                    │
│ 파일 트리│                                           │ 툴바 · 질문 · 답변 · 출처   │
│ (폴더/  │                                           │                            │
│  파일)  │                                           │                            │
│ 계층)   │                                           │                            │
├───┴───────────────────────────────────────────────┴────────────────────────────┤
│ Status Bar (선택): 마지막 저장 시각 · 오프라인 큐 · WebLLM 로딩 진행            │
└──────────────────────────────────────────────────────────────────────────────────┘
```

* **트리 ↔ 데이터 소스:** 로컬 폴더를 연 경우 **상대 경로 트리**를 1차로 보여 주고, **서버(MinIO/DB)** 에 반영된 목록과 **병합·정합**(업로드 대기·실패 표시). 로컬 미연결(클라우드만)이면 **서버에 등록된 경로**만으로 트리를 구성.
* **분할 비율:** 트리 | 에디터 | LoreDex 사이 **리사이즈 스플리터**; 기본 예: 트리 **~240px**, 나머지 **중앙·우 ≈ 55:45**. `localStorage` 등에 저장.
* **최소 폭(가이드):** 트리 ≥200px, 에디터 ≥400px, LoreDex ≥300px. 트리를 접으면 **에디터 | LoreDex** 만으로도 동작(과거 2열과 동일).

#### 3.1.2a 프로젝트 파일 브라우저(폴더 구조 뷰) 상세
* **목적:** "동기화된 파일이 트리의 어디에 있는지", LoreDex **출처 카드**의 원문 파일을 **열어 확인**하는 데 사용.
* **표시:** 폴더 **펼침/접힘**, 파일 노드에 **지원 확장자**·**색인 상태**(대기/완료/실패)·동기화 중 **스피너**. 비지원·미색인 파일은 시각적으로 구분.
* **에디터 연동:** 파일 **더블클릭**(또는 "열기") → 중앙 에디터에 로드. **현재 문서**는 트리에서 하이라이트. 다른 문서로 전환 시 **미저장 변경**이 있으면 확인.
* **LoreDex 연동:** 출처 **「파일에서 열기」** → 트리 해당 노드 포커스 + 에디터 오픈(장기: 청크 오프셋으로 스크롤).
* **(선택) 트리 검색:** 파일명·경로 부분 일치 필터로 대량 프로젝트 탐색 완화.

#### 3.1.3 App Bar (상단 바) 구성 요소
| 영역 | 내용 |
|------|------|
| 좌 | **로고** / 앱 이름 — 클릭 시 프로젝트 홈 또는 스튜디오 유지(정책에 따라) |
| 중 | **프로젝트 드롭다운** — 프로젝트 전환, 새 프로젝트(권한·티어에 따라) |
| 중·우 | **동기화·색인** — 아이콘 + 툴팁(유휴/업로드 중/색인 중) |
| 우 | **AI 엔진** — WebLLM / Gemini 전역 기본값 또는 현재 세션 표시 |
| 우 | **설정** · **프로필/로그아웃** |

#### 3.1.4 LoreDex 패널 내부 (우측 열)
* **상단:** 생성 엔진 선택(WebLLM·Gemini), Gemini 선택 시 **클라우드 전송** 배지.
* **입력:** 질문 입력창 + 전송(또는 Enter); **드래그로 채워진 키워드**가 있으면 입력창 상단에 칩으로 표시.
* **본문:** 사용자 질문 / AI 답변 **메시지 스트림**(스크롤 하단 고정); 답변 블록마다 **출처 카드** 접기/펼치기.
* **빈 상태:** "이 프로젝트 설정집을 읽고 질문해 보세요" + 색인 미완료 시 **색인 진행 중** 안내.

#### 3.1.5 Focus Editor 영역 (중앙 열)
* **문서 제목/회차** (선택): 에디터 상단에 한 줄 또는 블록 메타. **현재 열린 파일명**은 트리와 동기 표시.
* **본문:** Tiptap 캔버스 전체; 여백·폭 제한(가독성)은 `max-w-prose` 류로 중앙 정렬 가능.
* **자동 저장** 피드백은 App Bar 또는 하단 Status Bar에만 두고, 본문 플로우를 가리지 않음.

#### 3.1.6 보조 레이어 (모달·드로어)
* **설정:** 풀스크린 또는 우측 **드로어**(폭 ~400px) — 로그인·AI·프라이버시·테마.
* **Gemini 동의·쿼터 안내:** 모달(첫 사용 시).
* **프로젝트 생성/이름 변경:** 모달 또는 별도 페이지(단순 MVP는 모달).

#### 3.1.7 반응형·브레이크포인트 (원칙)
* **MVP 우선:** **데스크톱·태블릿 가로** (예: ≥1024px) 에서 **트리 + 에디터 + LoreDex** 3분할을 완전 지원.
* **좁은 화면:** **파일 트리**는 **접기 + 오버레이 패널** 또는 **하단 탭「파일」** 으로 분리. LoreDex도 **하단 시트** 또는 **탭(글쓰기 | 파일 | LoreDex)** 으로 전환.
* **WebLLM**은 연산·UI가 무거우므로 **모바일 세로**는 후순위로 두고, PRD에 "제한적 지원" 또는 "안내 메시지" 명시.

#### 3.1.8 기능 요구사항과의 매핑
* **F-12** 프로젝트 폴더·파일 트리: **3.1.2**, **3.1.2a**.

* **F-05~F-07** RAG·WebLLM·Gemini: **3.1.4** 우측 열 + **3.1.3** 엔진 표시.
* **F-09** 드래그 앤 인덱스: **3.1.4** 입력 칩 + 팝오버/인라인 카드.

### 3.2 온보딩: Google 로그인 → 프로젝트 생성
1. 랜딩에서 **Google로 계속하기** 클릭.
2. OAuth 콜백 후 백엔드가 사용자 레코드 생성·**JWT 발급**(액세스 짧게·리프레시 길게).
3. 첫 실행 시 **프로젝트 이름**·(선택) 로컬 폴더 연결 안내.

### 3.3 로컬 폴더 연동 (File System Sync)
* 사용자가 브라우저에서 `[내 문서/소설 설정집]` 폴더 접근을 허용하면, 지원 확장자(예: `.txt`, 추후 `.md`) 파일 목록·내용을 읽음.
* **폴더 구조 UI:** 동기화 결과는 항상 **§3.1.2a 프로젝트 파일 트리**에 반영된다. 사용자는 트리에서 **폴더 계층**을 탐색하고 파일을 골라 **에디터에서 연다**.
* **증분 동기화:** 파일 해시·mtime 기준으로 변경된 파일만 API로 업로드; 트리 노드별로 **동기화·색인 상태**가 갱신된다.
* 서버: **MinIO**에 원본 저장, 메타데이터(**상대 경로·부모 폴더 관계**)는 **PostgreSQL**에 기록하여 클라우드 전용 기기에서도 **동일 트리**를 재구성 가능하게 함.
* **Celery 태스크:** `ingest_file` → 텍스트 추출 → **LangChain** Text Splitter로 청크 → 임베딩 배치 → `pgvector` upsert.

### 3.4 메인 루프: Focus Editor + LoreDex Panel
* **프로젝트 파일 트리 (좌):** §3.1.2a — 열어둔 문서와 출처 탐색의 중심.
* **Focus Editor (중앙):** Tiptap 블록 에디터 — 단락·제목·목록, 자동 저장(로컬 draft + 주기적 서버 동기화).
* **LoreDex Panel (우):**
  1. 사용자가 자연어 질문 입력.
  2. **백엔드 RAG API:** 쿼리 임베딩 → `pgvector` 유사도 검색 → 상위 K개 청크 + 메타데이터(파일명, 경로) 반환.
  3. **생성 라우팅:** 사용자 설정이 **WebLLM**이면 프론트에서 프롬프트 조합 후 추론; **Gemini**이면 API 호출(키·엔드포인트 정책에 따름).
  4. 응답에 **인용 출처**(어느 설정 파일의 어느 문단 근처인지)를 함께 표시해 환각 검증에 도움.

### 3.5 드래그 앤 인덱스 (Drag & Index)
* 에디터에서 단어·구문 드래그 → 선택 이벤트로 키워드 추출.
* **짧은 지연 후** LoreDex에 자동 전달: `keyword_lookup` 또는 RAG `top_k=3` 만으로 요약 카드(팝오버/사이드 패널).
* 필요 시 "이 설정으로 채팅 열기"로 전체 질의로 확장.

### 3.6 작품 제작·집필 워크플로우 상세 (프로젝트 → 설정집 → 편/권 → AI 보조)

#### 3.6.1 프로젝트 생성·메타데이터
* **프로젝트**는 작품 단위 컨테이너다. 생성 시 최소 **작품 제목(표시명)** 을 받고, 선택적으로 **장르 태그**, **시점/시제 메모**, **외부 표기 규칙**(예: 인명 표기, 숫자 전각/반각)을 적을 수 있게 한다.
* **목적:** LoreDex 프롬프트·시스템 힌트에 **"이 작품은 OO 장르"** 를 넣어 답변 톤을 맞추고, 검색·필터 UI에서 프로젝트를 구분한다.
* **(선택) 표지·한 줄 소개** — 나중에 공개·퀵셀 때 쓸 메타; MVP는 생략 가능.

#### 3.6.2 소설 설정집 구성 (Lore Bible)
* **설정집**은 RAG의 **신뢰할 근거 레이어**다. 사용자는 다음을 **텍스트 파일**로 정리해 두는 것을 권장하고, 앱은 **폴더 트리·템플릿**으로 정돈을 돕는다.
  * **인물** — 이름, 별칭, 성격·말투 메모, 관계도 요약.
  * **세계관** — 지리, 역사, 마법/기술 규칙, 금기.
  * **용어·고유명사** — 철자·외래어 표기 통일표.
  * **줄거리·복선** — (선택) 타임라인 메모.
* **온보딩/빈 프로젝트:** `설정/` `원고/` 등 **추천 폴더 스켈레톤**을 한 번에 만들어 주거나, 마크다운 **템플릿 파일**을 복사해 넣을 수 있게 한다(로컬에서 편집 후 동기화).
* **역할 라벨(장기):** 파일 또는 청크에 `character` / `world` / `plot` 같은 **메타 태그**를 두면 RAG 필터(§4.3)와 연동 가능 — MVP는 **경로·파일명 규칙**으로 대체 가능.

#### 3.6.3 편·권·회차(연재 구조)와 원고 문서
* 웹소설은 **권·편·회(화)** 단위가 제각각이므로, 앱은 **강제하는 단일 모델**보다 **작가 네이밍 규칙 + 트리 정렬**을 우선한다.
* **권고 구조 예:** `원고/1권/001화.txt` … 또는 `원고/episodes/ch-001.md` — 트리에서 **회차 순서**가 보이면 충분.
* **현재 집필 문서:** 에디터 상단 또는 트리에서 **"지금 열린 회차 파일"** 이 명확해야 하며, LoreDex는 (선택) **「이 회차 파일」만** / **「설정집 + 직전 회차 원고」** 등 **검색 범위(scope)** 를 토글할 수 있게 한다(§3.6.5).
* **버전·백업:** 회차 파일은 MinIO·DB 메타로 **이력**을 남겨 실수 복구에 대비(티어·용량 정책과 연동).

#### 3.6.4 집필 중 ‘고민’ → 기능 매핑 (핵심)
작가가 멈추는 지점을 **기능**으로 쪼갠다. 구현은 대부분 **RAG + 프롬프트 프리셋 + WebLLM/Gemini** 조합으로 가되, 일부는 **전용 API**(예: scope 지정)가 유리하다.

| 작가 고민 | 제품이 하는 일 (요약) | 구현 포인트 |
|-----------|------------------------|-------------|
| **설정이 기억 안 남** | 드래그·질문으로 **내 설정집**에서 근거 인용 | RAG + 출처 카드 (기존) |
| **이전에 뭐라고 썼더라** | 동일 인물·용어의 **이전 기술** 찾기 | RAG, 파일 필터(인물명 파일 우선) |
| **말투가 캐릭터랑 안 맞는 것 같음** | 선택 구간 + 캐릭터 설정 청크로 **톤 점검·대안 초안** | 프리셋 프롬프트 + scope |
| **직전 회와 모순** | (선택) **직전 N회 원고**를 컨텍스트에 포함해 질문 | RAG scope: `settings` + `manuscript_recent` |
| **다음에 뭐 쓸지 막힘** | **줄거리만** 참고한 **아이디어 3안**(참고용, 비삽입) | 짧은 생성 + 면책 문구 |
| **용어 통일** | 동일 개념 다른 표기 **후보 나열** | RAG + 용어 파일 우선 |
| **한 장면 분량·호흡** | (선택) **요약·분할 제안**만 | Gemini 우선 검토(긴 문맥) |

* **주의 문구:** 자동 생성 텍스트는 **에디터에 바로 넣지 않고** 기본은 **복사/삽입 확인** 또는 **패널에만 표시**해 작가 의도를 해치지 않게 한다(§2.4).

#### 3.6.5 LoreDex 빠른 질문·프리셋 (Quick Actions)
* 자주 쓰는 질문을 **버튼/드롭다운**으로 제공해 입력 부담을 줄인다. 예시:
  * 「선택한 문장이 **[캐릭터 이름]** 말투와 맞는지」
  * 「**[키워드]** 가 설정집에 어떻게 정의되어 있지?」
  * 「직전 회차와 **시간·장소**가 이어지나?」(직전 원고 scope 켰을 때)
* **RAG 스코프 UI:** `설정집만` / `설정 + 현재 원고` / `설정 + 직전 회차 원고(1~3개)` — 기본값은 **설정집만**으로 두고, 연속성 질문 시에만 원고를 켠다(원고가 외부 유출 민감하면 로컬 WebLLM 경로 권장).

#### 3.6.6 설정집 품질·색인 상태
* **색인 완료 전**에는 LoreDex 상단에 **"설정집을 아직 다 읽는 중"** 을 띄워, 답이 빈약할 수 있음을 사전에 알린다.
* **추천 체크리스트(온보딩):** 인물 최소 1명, 세계관 규칙 1개 파일 등 — 강제는 아니나 **빈 RAG** 를 줄인다.

### 3.8 사용자 시나리오 (엔드투엔드)

#### 시나리오 A — 연재 전날, 설정 맞추기 (지민)
1. 브라우저로 앱 접속 → **Google 로그인**(이미 로그인한 동일 계정이면 세션 복원).
2. **프로젝트 "3부 외전"** 선택 → 좌측에 전회차 이어쓰기, 우측 LoreDex 대기.
3. 문장을 쓰다가 주인공 무기 이름이 헷갈림 → 단어 **드래그** → 우측에 **사전 카드**(RAG 상위 청크 요약 + 파일명).
4. 더 깊게 묻고 싶으면 카드에서 **「이 설정으로 질문하기」** → LoreDex 입력창에 키워드 프리필 후 자연어 질문.
5. 답은 기본 **WebLLM** — 민감한 대사 초안이 섞여도 외부 미전송. 긴 요약이 필요하면 **Gemini**로 전환 → **한 번 확인** 다이얼로그 → 응답 수신.
6. 원고는 **주기적 자동 저장** + (Pro면) **백그라운드 동기화** 표시만 보고 집필 지속.

#### 시나리오 B — 새 기기에서 이어쓰기 (Pro)
1. 새 PC에서 로그인 → **클라우드에 올려둔 프로젝트** 목록 표시.
2. 로컬 폴더는 아직 없음 → **웹에서만** 편집하거나, 나중에 **같은 폴더를 다시 연결**해 양방향 동기화(정책에 따라 충돌 해결 UX 필요 — 초기에는 "서버 우선" 또는 "최신 mtime" 단순 규칙 명시).

#### 시나리오 C — 웹툰 각본, 말투·외형 맞추기 (수진)
1. **프로젝트**에 캐릭터 시트·세계관·이전 화 시나리오를 폴더로 연동·동기화.
2. 대사를 쓰다가 주인공 **말투**가 헷갈림 → 해당 대사 **드래그** → LoreDex에 캐릭터 설정 청크가 뜸.
3. **"이 캐릭터가 이런 상황에서 쓸 법한 말투로 다듬어줘"**처럼 질문 — 기본 **WebLLM**, 길고 빠른 수정이 필요하면 **Gemini**로 전환.
4. (궁극 비전) 동일 프로젝트에서 나중에 **컷 리스트·이미지 생성** 파이프라인을 붙일 때, 지금 쌓인 원문·설정이 단일 소스가 됨.

### 3.9 UI·피드백 (상태 머신 요약)

## 4. 시스템 아키텍처 및 기술 스택 (Architecture)

### 4.1 논리 구조
```
[Browser]
  SvelteKit + Tiptap + WebLLM (WebGPU)
       │ HTTPS
       ▼
[nginx] ──► Static / SSR (frontend)
       └──► /api (Django + Gunicorn)
                 ├── PostgreSQL (+ pgvector)
                 ├── Redis (cache + Celery broker)
                 ├── MinIO (objects)
                 └── Celery workers (async jobs)
       │ (optional Gemini: FE direct or BE proxy)
       ▼
[Google OAuth] / [Gemini API]
```

### 4.2 스택
* **Frontend:** `SvelteKit`, `TailwindCSS`, `Tiptap`, `@mlc-ai/web-llm`(또는 선택 모델), Gemini 클라이언트 SDK 또는 REST.
* **Backend:** `Django`, `Django REST Framework`, `django-allauth` 또는 `social-auth` 계열로 **Google OAuth**, **`djangorestframework-simplejwt`** 등 JWT.
* **Async:** `Redis` 6+, `Celery` 5.x, (선택) Celery beat로 주기 작업.
* **Data:** `PostgreSQL` 15+, `pgvector`, `MinIO`(S3 호환).
* **AI 파이프라인:** `LangChain` 또는 경량 래퍼로 **splitter·embedding·vector store**; 임베딩 모델은 서버에서 실행하거나 외부 API(정책에 따라).

### 4.3 RAG 파이프라인 (상세)
1. **수집:** 업로드된 텍스트 또는 MinIO에서 텍스트 로드.
2. **전처리:** 인코딩 통일, 불필요 공백 제거(선택).
3. **청킹:** 문자 수·문단 구분 기준 splitter(예: 500~1000자, overlap 10~20%).
4. **임베딩:** 벡터 차원은 `pgvector` 컬럼과 일치.
5. **저장:** `project_id`, `file_id`, `chunk_index`, `content`, `embedding` 메타데이터.
6. **검색:** 쿼리 임베딩 후 유사도 상위 K, 필요 시 **MMR** 또는 필터(파일·태그).

### 4.4 인증·인가
* **Google OAuth:** authorization code flow, 백엔드에서 토큰 교환·사용자 식별.
* **JWT:** 액세스 토큰(짧음) + 리프레시 토큰(길음, 회전·폐기 정책 권장).
* **API:** `Authorization: Bearer <access>`; 모든 프로젝트·파일·RAG 엔드포인트는 **user_id** 기준 격리.

### 4.5 핵심 기술 심화 (Technical Deep Dive)

#### 4.5.1 WebLLM (`@mlc-ai/web-llm`)
* **역할:** 브라우저에서 **추론(생성)** 만 담당. 모델 가중치는 **첫 실행 시 캐시**(IndexedDB 등)에 남아 이후 로드 시간 단축.
* **런타임:** WebGPU 우선; 미지원 시 **CPU 경로**가 있더라도 체감 지연이 크므로 UI에서 **명시적 폴백**(Gemini 또는 "다른 브라우저 권장")을 둔다.
* **모델 선택:** 제품 초기에는 **소형~중형** Llama/Mistral 계열 등 **허용 라이선스** 모델을 고정하고, 모델 카탈로그 확장은 로드밴드·법무 검토 후.
* **프롬프트 구성:** 시스템 프롬프트에 **"주어진 인용 문맥만 근거로 답하고, 없으면 모른다고 말할 것"** 을 넣어 RAG 품질을 보조(서버와 동일한 철학).
* **리소스:** 탭 단위로 VRAM·메모리 압박 → **한 번에 하나의 큰 모델**, 백그라운드 탭에서는 추론 일시 정지 등 브라우저 정책 고려.

#### 4.5.2 Google Gemini API
* **호출 경로:** **백엔드 프록시**(권장) — API 키를 서버 환경에만 두고, 클라이언트는 세션·쿼터 검증 후 `/api/ai/gemini` 식으로 호출. **BYOK** 는 사용자 키를 서버에 암호화 저장 후 동일 프록시로 전달.
* **엔드포인트 개념:** `generateContent` 기반, **스트리밍**은 LoreDex에서 토큰 단위 표시 UX에 유리.
* **컨텍스트 조립:** RAG에서 가져온 청크는 **토큰 상한** 내에서 상위 K개만 포함; 초과 시 요약 또는 청크 축소.
* **세이프티·정책:** Gemini의 **safety category** 설정과 **사용 약관**(UGC, 창작물) 준수; 로그는 개인정보 최소화.

#### 4.5.3 RAG + pgvector
* **임베딩:** 쿼리·문서는 **동일 임베딩 모델·차원**을 사용. 예: 768/1536 차원 등 — `pgvector` 컬럼 차원과 **정확히 일치**.
* **거리 연산:** **코사인 거리** 또는 **inner product** 중 임베딩 모델 권장 방식에 맞춤; 인덱스는 데이터 규모에 따라 **HNSW** 또는 **IVFFlat**(파라미터는 벤치 후 고정).
* **검색 파이프라인:** `K`(예: 5~15) 후보 → (선택) **MMR** 로 다양성 확보 → **프로젝트·파일 ACL** 필터.
* **메타데이터:** 각 청크에 `source_file_id`, `offset`, `project_id` 를 넣어 UI **출처 클릭** 시 원문 위치로 스크롤(장기) 또는 파일명 표시(MVP).

#### 4.5.4 File System Access API (브라우저)
* **지원:** Chromium 계열 중심; **Safari/Firefox**는 지원 수준이 다르므로 "연결 불가 시 수동 업로드" 등 **대체 경로**를 PRD에 명시.
* **권한:** 사용자가 선택한 **디렉터리 핸들**은 세션 내 유지; 페이지 새로고침·재방문 시 **재승인** 필요할 수 있음 → UX 카피로 안내.
* **보안:** 앱은 **사용자가 고른 폴더만** 읽고, 서버로는 **업로드 API를 통한 데이터만** 전송(직접 디스크 경로 노출 없음).

#### 4.5.5 데이터 모델 (개략 스키마)
* **User** — `id`, Google `sub`, 이메일(표시용), 생성일.
* **Project** — `user_id`, 이름, 티어 스냅샷, `storage_used_bytes`.
* **FileObject** — `project_id`, MinIO **object key**, 원본명, `content_hash`, `mtime`, 색인 상태(`pending` / `ready` / `failed`).
* **Chunk** — `file_id`, `chunk_index`, `content`, `embedding vector`, (선택) 추후 `tsvector` 로 하이브리드 검색.
* **RefreshToken / Session** — JWT 리프레시 저장 시 해시만 저장 등 **OWASP 권고**에 맞춤.

#### 4.5.6 API 표면 (예시, REST)
* `POST /api/auth/google/callback` — OAuth 코드 교환.
* `POST /api/token/refresh` — 리프레시 토큰.
* `GET/POST /api/projects`, `GET/POST /api/projects/:id/files` — 메타·업로드 URL(프리사인 PUT) 등.
* `GET /api/projects/:id/files/tree` — **폴더 계층·파일 노드** 목록(경로, `file_id`, 색인·동기화 상태) — UI 트리 전용; 대량 시 **페이지네이션·지연 로딩(폴더 단위)** 검토.
* `POST /api/rag/query` — body: `{ project_id, query, top_k, scope }` — `scope`: `settings_only` | `settings_plus_current_manuscript` | `settings_plus_recent_manuscripts`(N) 등 §3.6.5와 정합.
* `POST /api/ai/gemini` — body: `{ project_id, messages, rag_chunks_ref }` — 서버에서 쿼터 검증 후 Gemini 호출.

#### 4.5.7 Redis·Celery 운영 디테일
* **브로커 URL** 과 **결과 백엔드**(선택: Redis에 결과 저장) 분리 여부는 트래픽에 따라 결정.
* **태스크 idempotency:** 동일 `file_id` + `content_hash` 재처리 시 기존 청크 삭제 후 재삽입 또는 upsert.
* **큐 분리:** `ingest` 큐와 `low_priority` 큐로 재색인·정리 작업 분리 가능.

#### 4.5.8 MinIO 객체 키 규칙 (예시)
* `users/{user_id}/projects/{project_id}/files/{file_id}/v{version}` — 버전 스냅샷 방식 또는 단일 최신 키 + DB 버전 관리.

## 4.7. Redis & Celery (작업 설계)

### 4.7.1 Redis 용도
* **Celery broker** (또는 Redis + 다른 broker 조합은 팀 표준에 따름; 단일 Redis는 단순화에 유리).
* **캐시:** 세션·RAG 메타·레이트 리밋 카운터.
* **짧은 TTL 캐시:** 동일 질의에 대한 임베딩 캐시(선택).

### 4.7.2 Celery 태스크 예시
| 태스크 | 설명 | 우선순위 |
|--------|------|----------|
| `files.process_upload` | 업로드 완료 후 청크·임베딩·pgvector 저장 | P0 |
| `files.reindex_project` | 프로젝트 전체 재색인 | P1 |
| `files.delete_vectors` | 파일 삭제 시 벡터 정리 | P0 |
| `notifications.send` | (선택) 이메일·웹훅 | P2 |
| `maintenance.cleanup` | (beat) 오래된 임시 세션·로그 | P2 |

* **재시도:** 네트워크·임베딩 API 실패 시 exponential backoff.
* **모니터링:** Flower(선택), Celery 로그 + Sentry 등.

## 4.8. Gemini & WebLLM 혼용 정책 (구현 가이드)

* **UI:** LoreDex 상단에 **생성 엔진: [WebLLM ▼] [Gemini]** 토글. Gemini 선택 시 **한 번 더 확인** 또는 설정에 "항상 허용" 저장.

### 5.1 티어 (초안)
| 항목 | Basic (무료) | Pro (구독/평생) |

| 프로젝트 수 | 1 | 무제한(합리적 상한은 운영 정책) |

## 6. 비기능 요구사항 (NFR)

* **성능:** RAG 검색 p95 응답 목표(예: 500ms 이하, 데이터량에 따라 조정), 에디터 타이핑 지연 없음.
* **가용성:** 프로덕션 DB·MinIO 백업·복구 절차 문서화.
* **보안:** HTTPS 전 구간, 비밀키·OAuth 클라이언트 시크릿 환경변수, MinIO 버킷 정책(프로젝트 단위 prefix).
* **규정:** Gemini·Google 로그인 사용 시 각 **이용약관·개인정보 처리방침**에 서비스 내용 반영.

---

## 7. 마일스톤 (Milestones)

* **Phase 1 (MVP - Local First):** SvelteKit 분할 UI, Tiptap, WebLLM 로드·추론, Docker Compose(프론트·백·DB·Redis 최소), nginx 로컬 프록시.
* **Phase 2 (RAG & Auth):** Google OAuth, JWT, MinIO 업로드 API, Celery 색인 파이프라인, pgvector RAG, Gemini API 연동(백엔드 프록시 권장).
* **Phase 3 (Integration):** File System Access, 드래그 앤 인덱스, WebLLM/Gemini 전환 UX·쿼터, 프로덕션 Docker + nginx + TLS.
* **Phase 4 (Launch):** CBT, 텀블벅 등 초기 유저, 정식 런칭, 모니터링·알림 정착.
* **Phase 5 (North Star — 텍스트→웹툰, 별도 기획):** 원문·설정을 구조화된 **컷/씬 단위**로 분해, 스토리보드·캐릭터 일관성을 위한 **이미지 생성 파이프라인**(모델·저작권·편집 UX) 설계. Phase 1~4에서 확보한 **단일 프로젝트 소스**를 입력으로 삼는다.

## 8. 리스크·오픈 이슈

* **WebGPU 미지원 브라우저:** WebLLM 불가 시 Gemini만 사용하거나 안내 메시지.

## 9. 용어
* **RAG:** Retrieval-Augmented Generation — 검색된 문맥을 생성 프롬프트에 붙여 답변 품질을 높이는 방식.
* **LoreDex:** 로어(설정)를 색인해 검색·질의하는 우측 패널 제품명.
* **BYOK:** 사용자가 자신의 API 키를 등록해 쓰는 방식.
* **텍스트→웹툰 (North Star):** 집필된 스토리·대사·설정을 바탕으로 웹툰 제작에 필요한 **컷 구성·비주얼 생성·편집**까지 이어 주는 기능군. 본 PRD의 **후속 Phase**에서 스펙을 확정한다.
